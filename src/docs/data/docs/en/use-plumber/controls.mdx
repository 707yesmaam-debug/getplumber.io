Plumber allows defining controls for GitLab projects, covering both CI/CD configuration and project settings. When a control is not respected, an issue is created.

## CI/CD Container Images
| Control | Problem | Impact |
|---------|---------|--------|
| Containers images must come from authorized sources [ISSUE-2](/docs/use-plumber/issues/ISSUE-2) | <span style={{fontSize: '0.85em'}}>Verifies that container images used to run your CI/CD pipelines come from authorized and trusted sources.</span> | <span style={{fontSize: '0.85em'}}>Helps mitigate security risks introduced by the use of malicious, compromised, or vulnerable images.</span> |
| Container images must not use forbidden tags [ISSUE-3](/docs/use-plumber/issues/ISSUE-3) | <span style={{fontSize: '0.85em'}}>Verifies that container images used to run your CI/CD pipelines rely on authorized tags.</span> | <span style={{fontSize: '0.85em'}}>Helps mitigate both security and functional risks introduced by the use of unverified, outdated, or compromised image versions.</span> |
| Container images must be pinned by digest [ISSUE-33](/docs/use-plumber/issues/ISSUE-33) | <span style={{fontSize: '0.85em'}}>Verifies that container images are referenced by their SHA256 digest rather than a mutable tag.</span> | <span style={{fontSize: '0.85em'}}>Prevents supply chain attacks where a tag is overwritten with a compromised image, guaranteeing exact image content in every pipeline run.</span> |

## CI/CD Variables
| Control | Problem | Impact |
|---------|---------|--------|
| CI/CD variables must be protected [ISSUE-4](/docs/use-plumber/issues/ISSUE-4) | <span style={{fontSize: '0.85em'}}>Verifies that CI/CD variables used in a project have the protected field enabled.</span> | <span style={{fontSize: '0.85em'}}>Ensures sensitive values are restricted to protected branches or tags, reducing unauthorized exposure.</span> |
| CI/CD variables must be masked [ISSUE-5](/docs/use-plumber/issues/ISSUE-5) | <span style={{fontSize: '0.85em'}}>Verifies that CI/CD variables used in a project have the masked field enabled.</span> | <span style={{fontSize: '0.85em'}}>Prevents variable values from being exposed in pipeline logs, reducing the risk of leaks.</span> |
| Pipeline must not enable debug trace [ISSUE-34](/docs/use-plumber/issues/ISSUE-34) | <span style={{fontSize: '0.85em'}}>Verifies that `CI_DEBUG_TRACE` and `CI_DEBUG_SERVICES` are not enabled in the pipeline configuration.</span> | <span style={{fontSize: '0.85em'}}>Prevents exposure of all CI/CD variable values in job logs, including secrets and tokens.</span> |
| Pipeline must not use unsafe variable expansion [ISSUE-35](/docs/use-plumber/issues/ISSUE-35) | <span style={{fontSize: '0.85em'}}>Detects user-controlled CI variables expanded in shell re-interpretation contexts (`eval`, `sh -c`, `bash -c`, `source`).</span> | <span style={{fontSize: '0.85em'}}>Prevents command injection via crafted branch names, MR titles, or commit messages (OWASP CICD-SEC-1).</span> |

## CI/CD Secrets
| Control | Problem | Impact |
|---------|---------|--------|
| Pipeline configuration must not contain secrets [ISSUE-7](/docs/use-plumber/issues/ISSUE-7) | <span style={{fontSize: '0.85em'}}>Uses Gitleaks to verify that both merged and unmerged CI/CD configurations don't have leaked secrets.</span> | <span style={{fontSize: '0.85em'}}>Prevents exposure of API keys, passwords, or tokens that could lead to unauthorized access.</span> |

## Pipeline Composition
| Control | Problem | Impact |
|---------|---------|--------|
| Pipelines must include templates [ISSUE-12](/docs/use-plumber/issues/ISSUE-12) [ISSUE-13](/docs/use-plumber/issues/ISSUE-13) | <span style={{fontSize: '0.85em'}}>Verifies that the projects contain specific templates. This control can also allow overriding certain variables in the included templates.</span> | <span style={{fontSize: '0.85em'}}>Ensures pipelines comply with required security and compliance practices.</span> |
| Pipelines must include components [ISSUE-29](/docs/use-plumber/issues/ISSUE-29) [ISSUE-30](/docs/use-plumber/issues/ISSUE-30) | <span style={{fontSize: '0.85em'}}>Verifies that the projects contain specific GitLab components. This control can also allow overriding certain variables in the included components.</span> | <span style={{fontSize: '0.85em'}}>Ensures pipelines integrate mandatory security and compliance steps.</span> |
| Pipeline must include required phases [ISSUE-25](/docs/use-plumber/issues/ISSUE-25) | <span style={{fontSize: '0.85em'}}>Verifies that the CI/CD pipeline includes a group of job types.</span> | <span style={{fontSize: '0.85em'}}>Ensures completeness and compliance of the pipeline execution flow.</span> |
| Pipeline must not contain hardcoded jobs [ISSUE-8](/docs/use-plumber/issues/ISSUE-8) | <span style={{fontSize: '0.85em'}}>Verifies that no hardcoded job is used in CI/CD pipelines.</span> | <span style={{fontSize: '0.85em'}}>Improves maintainability and ensures compliance with best practices.</span> |
| Pipeline must not override template jobs [ISSUE-9](/docs/use-plumber/issues/ISSUE-9) | <span style={{fontSize: '0.85em'}}>Verifies that no job coming from an included template is overridden in the project CI/CD configuration.</span> | <span style={{fontSize: '0.85em'}}>Prevents inconsistent or insecure pipeline configurations caused by job overrides.</span> |
| Pipeline must not use forbidden ref in includes [ISSUE-11](/docs/use-plumber/issues/ISSUE-11) | <span style={{fontSize: '0.85em'}}>Verifies that the included refs are using specified tags.</span> | <span style={{fontSize: '0.85em'}}>Prevents reliance on insecure or non-compliant references.</span> |
| Pipeline must use only up-to-date includes [ISSUE-10](/docs/use-plumber/issues/ISSUE-10) | <span style={{fontSize: '0.85em'}}>Verifies that the included pipelines are up-to-date compared to their source.</span> | <span style={{fontSize: '0.85em'}}>Reduces risks from outdated or vulnerable templates.</span> |

## Access and Authorization
| Control | Problem | Impact |
|---------|---------|--------|
| Branch must be protected [ISSUE-14](/docs/use-plumber/issues/ISSUE-14) [ISSUE-27](/docs/use-plumber/issues/ISSUE-27) | <span style={{fontSize: '0.85em'}}>Verifies that the project configuration respects the protection, push, merge and owner approval on included branch names.</span> | <span style={{fontSize: '0.85em'}}>Prevents unauthorized modifications and enforces branch protection standards.</span> |
| MR approval rules must have at least N approval required [ISSUE-22](/docs/use-plumber/issues/ISSUE-22) | <span style={{fontSize: '0.85em'}}>Verifies that the project merge request approval rules that cover all protected branches have a minimum number of approval requirements.</span> | <span style={{fontSize: '0.85em'}}>Prevents unreviewed code from being merged, reducing security risks.</span> |
| MR approval settings must be compliant [ISSUE-23](/docs/use-plumber/issues/ISSUE-23) | <span style={{fontSize: '0.85em'}}>Verifies that MR approval settings are properly configured.</span> | <span style={{fontSize: '0.85em'}}>Ensures compliance with review and security requirements.</span> |
| An MR approval rule must be defined to cover all protected branches [ISSUE-24](/docs/use-plumber/issues/ISSUE-24) | <span style={{fontSize: '0.85em'}}>Verifies that the protected branches have at least one approval rule.</span> | <span style={{fontSize: '0.85em'}}>Ensures protected branches cannot bypass review processes.</span> |
| MR settings must be compliant [ISSUE-28](/docs/use-plumber/issues/ISSUE-28) | <span style={{fontSize: '0.85em'}}>Verifies that the project's merge request settings are correct in terms of merge method, resolving differences, squashing, etc.</span> | <span style={{fontSize: '0.85em'}}>Reduces risk of unauthorized or insecure code changes.</span> |
| Number of project members must respect a quota [ISSUE-31](/docs/use-plumber/issues/ISSUE-31) | <span style={{fontSize: '0.85em'}}>Verifies that the project configuration respects the owner, maintainer and developer quotas.</span> | <span style={{fontSize: '0.85em'}}>Prevents uncontrolled access that could weaken project security.</span> |
| Number of group members must respect a quota [ISSUE-32](/docs/use-plumber/issues/ISSUE-32) | <span style={{fontSize: '0.85em'}}>Verifies that the group configuration respects the owner, maintainer and developer quotas.</span> | <span style={{fontSize: '0.85em'}}>Prevents uncontrolled access at group level, strengthening governance.</span> |

| Control | Problem | Impact |
|---------|---------|--------|
| Project must have a security policy source [ISSUE-20](/docs/use-plumber/issues/ISSUE-20) | <span style={{fontSize: '0.85em'}}>Verifies if the projects have a specific project as their source of security policy.</span> | <span style={{fontSize: '0.85em'}}>Ensures compliance with security policy and reduces risk of unmanaged vulnerabilities.</span> |

## Pipeline Security
| Control | Problem | Impact |
|---------|---------|--------|
| Security jobs must not be weakened [ISSUE-36](/docs/use-plumber/issues/ISSUE-36) | <span style={{fontSize: '0.85em'}}>Detects security scanning jobs (SAST, Secret Detection, Container Scanning, etc.) weakened by `allow_failure: true`, `rules:` overrides with `when: never`/`when: manual`, or `when: manual` at job level.</span> | <span style={{fontSize: '0.85em'}}>Prevents silently neutralized security scans that give a false sense of compliance (OWASP CICD-SEC-4).</span> |
